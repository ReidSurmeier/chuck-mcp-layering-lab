# OPRO — Large Language Models as Optimizers

**arXiv 2309.03409** (Sep 2023, Google DeepMind) — Yang, Wang, Lu, Liu, Le, Zhou, Chen.

## Why this matters for chuck-mcp v2

OPRO is the *opposite* pattern from what chuck-mcp v2 needs. In chuck-mcp v2 the LLM is a **one-shot translator** (prompt → structured constraints, then hand off to solver). OPRO is an **iterative optimizer** where the LLM proposes solutions, sees their loss, and refines. **OPRO is the wrong pattern for chuck-mcp v2 V1.** Document for future reference.

## What OPRO does

LLM gets a meta-prompt containing:
1. Problem description in natural language
2. Past (solution, score) trajectory pairs, sorted ascending by score
3. Instructions to propose a new solution that scores better

LLM emits a new candidate. Caller evaluates it, appends to trajectory, loops. With ~20 iterations of GPT-4, OPRO beats hand-designed prompts on GSM8K by ~8% and on BBH by up to 50%.

## Demonstrated tasks

- Linear regression (`y = wx + b`, find `w, b`): LLM converges to optimum within 10–20 iterations
- Traveling salesman (small instances n≤50): LLM gets within a few % of optimal
- **Prompt optimization** (the headline application): "Let's think step by step" → "Take a deep breath and work on this problem step-by-step" beats it by 8% on GSM8K

## Implications for chuck-mcp v2

**Right now: don't use OPRO pattern.** chuck-mcp v2 V1 does one-shot translation prompt → constraints, then JAX L-BFGS-B does the actual optimization. The solver is gradient-based and dramatically faster than an LLM-in-loop.

**Future v2.5 / v3 candidate:** OPRO-style outer loop where the LLM gets to see (proposed_constraints, solver_residual_dE) tuples and refine the constraints. Use when:
- Calibration off (LLM picked the wrong pigment family — the dE residual is structured, not noise)
- User wants to iterate without re-typing prompt
- We can spare 5–20 minutes for an outer loop

This matches the v2 design's "ambiguity flagging" mechanism: when the LLM is unsure ("warm-tonal — assumed YR hue family"), v3 could turn that into an OPRO-style refinement loop where the user only needs to confirm or correct, never re-type.

## Meta-prompt structure (canonical)

```
You are helping me <task description>.
Below are some <solutions> and their <scores>. The <solutions> are arranged in
ascending order based on their <scores>, where higher <scores> are better.

<solution1>
<score1>
<solution2>
<score2>
...
<solutionN>
<scoreN>

Generate a <solution> that is different from all <solutions> above, and has a
<score> as high as possible.
```

Trajectory is bounded (e.g., last 20 entries). Solutions are sorted by reward.
